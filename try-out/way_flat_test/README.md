# 平面形状（道路・線路）のBlenderレンダリング検証

作業日: 2026-07-15

## 目標
これまでの立体物(box)ベースのアドオン生成（駅拡張建物・0系新幹線等）とは異なり、
`obj=way` 系の平面に近い形状で同じBlenderパイプラインが通用するか確認する。
- 道路: 高さ0の完全な平面
- 線路: 道床(バラスト)+レール分だけわずかに盛り上がる薄い形状

## 結果
達成。Blenderレンダリング→dat作成→makeobj pak化→addonsフォルダ配置→ゲーム内表示までの
一連のパイプラインが「平面に近い形状(道路・線路)」でも通用することを確認できた。
ユーザーによる実機フィードバックを12周（表示位置のズレ→寸法の太さ→レール個別の
高さ微調整→Z下げによる奥側レールの消失バグ修正→タイル境界のアンチエイリアス半透明化→
半透明化解消後に境界に出た色みの差→進行方向の継ぎ目ズレ→レールが道床に埋もれて
見える段差消失→pak128デフォルト線路と接続時の奥側ズレ→奥のレールが道床から
ぶら下がって見える不具合→道床の段差自体を無くしたいという要望→Xシフトのみだと
横方向にずれすぎる）反映し、道路・線路とも最終的に「地面・デフォルト線路と正しく
位置が揃う」ことをユーザー実機確認済み。ただし今回検証したのは1方向のみ（4方向とも
同一画像を使い回す簡略構成）で、8方位すべてを個別に調整するコストが大きいことが
判明したため、そこは今回のPoCのスコープ外として区切り、ここで完了とする。

## 試したこと
- `refs/pak128/infrastructure/roads/road_030.dat` / `rail_tracks/rail_100_tracks.dat` を参照し、
  `obj=way` の実データ構造を調査（`Image[方向][season]` キー形式、`icon=> file.row.col` の
  `>` 接頭辞はズーム非対応指定であることを`image_writer.cc`で確認）
- `refs/simutrans/.../way_writer.cc` を読み、`obj=way` の必須項目は `image[-][0]` のみで
  他の方向キーが空でも fatal にならないことを確認（未対応方向は単に描画されないだけ）
- 道路: `primitive_plane_add`（厚み0）+ センターライン用の別平面をz=0.001だけ持ち上げて重畳
  （z-fighting回避。実質的な厚みとしては無視できる量）
- 線路: 道床(バラスト)を薄いcube、その上にレール2本をさらに薄いcubeで重ねる3パーツ構成
  - 高さは「1フロア相当(0.816BU)」に対する割合(`HEIGHT_FRAC`)で指定し、
    isometric高さ補正係数`√6/3`を掛けてBlender単位に変換する方式にした
    （道床10% + レール3% → 実寸で道床0.0816BU、レール0.0245BU、合計高さ約0.106BU）
- 4方向とも同一レンダリング画像を使い回す簡略化（station_test/plateau_buildingの前例踏襲、
  roadmap.mdのTODO「4方向で異なる画像を使う」は本PoCでも未対応のまま）
- `dat_linter lint .` を実行 → 出力なし（現状の dat_linter は `obj=building` のみ対応のため
  `obj=way` は検証対象外。roadmap.mdのTODO「vehicle本体/way など他obj種別のlinter対応」に該当）
- `makeobj VERBOSE DEBUG pak128 ... road.dat` で画像サイズを確認 → `128x128`で警告・エラーなし
- `road_flat.pak` / `rail_flat.pak` をそれぞれ `simuwin/addons/pak128/` に配置
- ユーザーによる実機確認: ゲーム内に表示はされたが、画像がタイル上空に浮いて見える
  （目視で約32px下にずれている）と報告を受ける
- 原因調査: カメラの注視点(target)のZ座標を、道路は`0`（地面ぴったり）、線路は
  `rail_top_z/2`（オブジェクト自身の中心）と、オブジェクトごとに異なる値で設定していた。
  一方、動作確認済みのstation_cube(高さ0.816のbox)は`target.z=0.408`固定
  だったことに気づき、「target.zはオブジェクトの高さに関係ない固定の地面アンカー」
  という仮説を立てて検証（詳細は[knowledge.md](../../knowledge.md)参照）
  - 観測されたズレ32pxを`px_shift = world_z_shift × 0.866 × (128/ortho_scale)`の式に
    当てはめると`world_z_shift ≈ 0.408`となり、まさに station_cube の target.z と一致
- 4スクリプトすべての target.z を `0.408` 固定に修正し、再レンダリング→pak再ビルド→
  addonsフォルダへ再配置
- 表示位置修正後の再確認で、寸法についてユーザーから4点の指摘を受ける
  - 道路: センターラインが太め（半分でよい）
  - 線路: 道床(バラスト)の高さが高すぎる（pak128実物は約1px程度）
  - 線路: レールの高さ(太さ)が高すぎる（pak128実物は約1px程度）
  - 線路: レール幅(ゲージ)が広すぎる（pak128実物はEW方向で画面横方向に約10px）
- 「1フロア相当に対する割合」というBU単位の指定方法では目視のpx単位フィードバックに
  正確に応えられないため、カメラ変換係数を実測してpx→BUの変換式を導出する方式に変更。
  同じカメラ設定(ortho_scale=√2, rotation=(60°,0,45°))でcam.matrix_world.to_3x3()の
  right/up基底ベクトルを出力するprobeスクリプトを一時的に実行し、以下を確認:
  - 世界Z軸1BU分のオフセット → 画面縦方向78.38px（`(128/√2)×cos30°`）
  - 世界X軸(またはY軸)1BU分のオフセット → 画面横方向64.0px（`(128/√2)×cos45° = 128/2`、
    ちょうど半分になるきれいな数値）
- 上記の変換係数を使い、道床・レールの高さをそれぞれ「1px相当のBU値」で直接指定、
  レールの間隔(ゲージ)を「10px相当のBU値」で直接指定する方式に書き直し
  （`HEIGHT_FRAC`のような相対割合ではなく`BALLAST_HEIGHT_PX`等のpx単位の定数に変更）。
  センターラインの幅は単純に半分(`0.04`→`0.02`)に変更
- 再レンダリング→pak再ビルド→addonsフォルダへ再配置
- ユーザーによる再確認: 道路はOK。線路は「手前側の軌道は1px下げ、奥側の軌道は3px下げ
  ぐらいでちょうどよい」と指摘を受ける（2本のレールそれぞれに異なる追加補正が必要、
  一律のシフトでは対応できない）
- カメラのright/up基底ベクトル実測（probeスクリプトの結果、knowledge.md参照）から、
  「world +X オフセットは画面下方向(手前)に、-Xオフセットは画面上方向(奥)に動く」
  ことが分かっていたため、+X側のレールを手前・-X側を奥とみなし、それぞれ個別に
  `z -= extra_down_px / PX_PER_BU_Z` で追加のZ下げ補正を加える方式に変更
  （手前1px相当≈0.0128BU、奥3px相当≈0.0383BU）。2本のレールを同一の`for`ループで
  生成する実装から、`(offset_x, name, extra_down_px)`のリストで個別指定する実装に変更
- 再レンダリング→pak再ビルド→addonsフォルダへ再配置
- ユーザーによる再確認: 手前側の位置はOK。ただし奥側の線路が表示されていない（消失）
- 原因調査: 奥側の追加下げ量3px(≈0.0383BU)が、レール自体の高さ(≈0.0128BU)を超えていた。
  Zをその分減らすと奥側レールの上面が道床(バラスト, 高さ0.0128BU)の下面(z=0)より
  さらに下に潜り込み、不透過なバラストの真下に完全に隠れて画面上で非表示になっていた
  （元々ちょうど「バラスト上面ぎりぎり」の高さで置いていたため、余裕が全くなかった）
- 修正方針: Z(高さ)を変えずに画面上の縦位置だけを動かす方法に変更。カメラのright/up
  基底ベクトル実測値（world+Xは画面(dx=+64,dy=-32)/BU、world+Yは(dx=+64,dy=+32)/BU）
  を使い、world X,Yを逆符号で同量動かすと画面横方向(dx)の変化を打ち消しながら
  画面縦方向(dy)だけを動かせることを導出（`dx=64a+64b=0 → b=-a`,
  `dy=-32a+32b=-64a=-n_px → a=n_px/64`）。この`screen_down_shift_xy()`関数で
  手前・奥両方のレールのオフセットを計算し直す実装に変更（Zは道床上面の元の高さのまま）
- 再レンダリング→pak再ビルド→addonsフォルダへ再配置
- ユーザーによる再確認: 位置は揃った(OK)。ただし「タイルの周囲1pxぐらいがアンチエイリアスの
  影響か半透明になっていて、連続設置すると隙間ができる」と指摘を受ける
- 原因: 平面(道路)の外周がちょうどタイル境界と一致する形状のため、Blenderのアンチエイリアス
  でその境界が部分透明(半透明)のピクセルとしてレンダリングされていた。Simutrans側は
  隣接タイルとの合成を単純に行うため、半透明ピクセルからは下の地面(芝生等)がうっすら
  透けて見え、タイルを連続設置すると継ぎ目に薄い隙間があるように見えていた
- 対策: 4倍解像度(512x512、`SUPERSAMPLE`定数)でレンダリングしてから128x128に縮小し、
  縮小後にアルファチャンネルを閾値128で0/255の二値にハード処理する`postprocess.py`
  （PIL使用、Blender外の素のPythonで実行）を追加。半透明ピクセルを一切残さない
  ことで、タイル境界のアンチエイリアスによる隙間を解消する
- 4スクリプトすべてを「512x512の"_raw.png"を出力→postprocess.pyで128x128+アルファ二値化」
  の2段構成に変更。makeobj向けの最終ファイル名(road_flat.png等)は変わらないため
  road.dat/rail.dat自体は変更不要
- 再レンダリング→postprocess→pak再ビルド→addonsフォルダへ再配置
  （road_flat.png/rail_flat.pngのアルファ値が`{0, 255}`の2値のみになっていることを
  `python3 -c "Image.open(...).split()[3]"`で確認）
- ユーザーによる再確認: 半透明部分は解消。ただし「境目部分は色みの差が出ている」と
  新たな指摘を受ける
- 原因調査(1つ目): road_flat_raw.pngの境界付近を直接ピクセル調査すると、内部の均一な
  色(123,124,128)に対し境界の1px内側だけ明るい(129,130,134)になっていた。透明領域の
  RGBがBlenderの背景色である黒(0,0,0)のまま保存されており、LANCZOS縮小時に
  「不透明な物体色」と「透明領域の黒」を混ぜてしまうことでオーバーシュート
  （リンギング）が発生していたと判明
- 対策(1回目): 縮小前に透明領域のRGBを最も近い不透明ピクセルの色で塗りつぶす
  color bleed/dilate処理をnumpyで実装（8近傍平均を複数回繰り返すシンプルな拡張）。
  しかし適用してもフリンジは変わらず再発（原因調査を継続）
- 原因調査(2つ目): RGBを完全に均一化してから`Image.resize()`をRGBAのまま
  一括で呼んでも直らないことを確認。試しにRGB(3ch)とアルファ(1ch, "L"モード)を
  完全に別画像としてresizeしたところフリンジが消えたため、PillowのRGBA画像に対する
  `resize()`がアルファを考慮した内部処理を行っており、それがフリンジの原因だったと
  特定（RGBを塗りつぶすだけでは、RGBAとして一括resizeする限り再発する）
- 対策(2回目、最終的な修正): color bleed済みのRGBを3chの"RGB"画像として、元の
  アルファチャンネルを1chの"L"画像として、それぞれ独立にresizeしてから最後に
  マージする実装に変更。境界付近のRGBが内部と同じ値のまま保たれることを確認
- 再レンダリング不要（`_raw.png`はそのまま）、postprocess.py再実行→pak再ビルド→
  addonsフォルダへ再配置
- ユーザーによる再確認: 色みの差はOK。ただし「線路の軌道位置が車両進行方向基準で
  ずれていて連続設置すると隙間ができている」と新たな指摘を受ける
- 原因: 手前/奥レールの追加下げ補正で使っていた`screen_down_shift_xy()`が
  world Yも動かしていた。レールはY方向にscale.y=1.0（タイル境界ぴったり）で
  作っていたため、Yオフセットが加わるとレールの実際のY方向の footprint がタイル
  境界からずれ、進行方向に連続設置したときに継ぎ目に隙間ができていた
  （タイル境界の半透明化とは別の、Y軸方向の話）
- 対応方針の検討: 「Yを動かさない」「道床/レールの高さは1pxのまま変えない」の
  両方を満たしたまま「手前1px・奥3px」の差を再現しようとすると、Zのみでの
  下げ量は道床上面が上限（1px）のため奥の3pxが物理的に不可能、Xのみでは
  すでに承認済みのゲージ(10px)が変わってしまうというトレードオフに突き当たった。
  ユーザーに確認し、「継ぎ目を優先し、手前奥とも同じ高さ(1px)に揃える」方針を選択
- 実装: `screen_down_shift_xy()`とX/Yオフセットを撤廃し、Yは常に0のまま、
  手前・奥のレールを同一の`RAIL_EXTRA_DOWN_PX=1.0`のZシフトのみで統一
- 再レンダリング→postprocess→pak再ビルド→addonsフォルダへ再配置
- ユーザーによる再確認: 継ぎ目はOK。ただし「線路が道床と同じ高さになってしまった」
  との指摘を受ける。レールを`RAIL_EXTRA_DOWN_PX=1.0`分下げた結果、レールのZ範囲が
  道床のZ範囲(0〜1px)とちょうど一致してしまい、道床から突き出て見えるはずの段差が
  消えていた（数値上は把握していたが、見た目の影響を軽視していた）
- ユーザーから「立体的な正しさよりレンダリング後の正確さが必要」という方針を受け、
  レール側を動かさず、道床だけを追加で1px下げる(`BALLAST_EXTRA_DOWN_PX=1.0`)方式を
  採用。道床が実質的にz<0（地面のやや下）に沈む3D的には不自然な配置になるが、
  最終的な2Dスプライトの見た目でレールが道床から1px突き出て見えれば目的は達成される
  ため、3D空間上の物理的な整合性より優先する
- 再レンダリング→postprocess→pak再ビルド→addonsフォルダへ再配置
- ユーザーによる再確認: 段差はOK。ただし「pak128デフォルトの線路とつなぐとやはり
  奥側の線路がずれてしまう」と新たな指摘。実物の`rail_100_tracks.png`を確認すると、
  道床全体を覆う枕木の上に細いレールが乗る構造で、うちの単純な2本線モデルとは
  道床幅・レール位置とも大きく異なることが判明。ピクセル単位で実物を精密測定して
  完全一致させる方法もあるが工数が大きいため、ユーザーに方針を確認。
  「ずれの量は修正前と同じ3pxなので、それを再現できないか試してほしい」との回答
- 対応: 道床を1px下げたことで、道床のZ範囲が`[-1px, 0px]`に変わっていた。
  以前(道床が`[0px,1px]`にあった頃)は奥のレールを3px下げると道床の範囲に潜り込んで
  隠れたが、今は道床が下にずれているため、奥のレールを3px下げても道床の範囲
  (`[-1px,0px]`)とは重ならず(新しい範囲`[-3px,-2px]`)、隠れずに描画されることに気づいた。
  手前は`NEAR_EXTRA_DOWN_PX=1.0`のまま、奥だけ`FAR_EXTRA_DOWN_PX=3.0`に戻す形で、
  Yには触れずZのみで手前1px/奥3pxの非対称な下げを復元
- 再レンダリング→postprocess→pak再ビルド→addonsフォルダへ再配置。奥のレールが
  道床から切り離れて浮いたような3D配置になるが、目視でどちらのレールも隠れずに
  描画されることを確認
- ユーザーによる再確認: 「奥側の線路消えっちゃってるね」と報告。ローカルで
  `rail_flat.png`のピクセルを調査すると奥のレールのデータ自体は画像内(y=81〜122)に
  存在し、makeobjのクロップ範囲(11,70)-(116,122)にも収まっていたため、消えている
  というより見た目に問題があるはずと判断し、詳細を尋ねる
- ユーザーから「`rail_flat_raw.png`を見ると奥の線路は道床の下側にある」と回答。
  座標を計算し直すと、奥のレールのZ範囲`[-3px,-2px]`の上端(-2px)が、手前を支える
  道床のZ範囲`[-1px,0px]`の下端(-1px)と重ならず**さらに下**にあり、道床から
  ぶら下がって浮いているように見える配置になっていたと判明（道床の下に隠れて
  「消える」のではなく、道床から離れて「ぶら下がって」見えていた）
- 対応: 道床を単一のboxから「手前を支える半分」「奥を支える半分」の2つのboxに分割し、
  各半分をそれぞれの真上のレールと同じ量(手前1px, 奥3px)だけ沈める。これにより
  奥のレールも、自分の真下にある(奥用に沈めた)道床から1px突き出て見えるようになり、
  手前と同じ「道床から1px突き出た段差」の見た目を奥側でも再現できる
- 再レンダリング→postprocess→pak再ビルド→addonsフォルダへ再配置
- ユーザーによる再確認: デフォルト線路との位置合わせはOK。ただし「道床の位置を
  手前と揃えられないか」と新たな要望。道床を手前/奥に分割して段差をつけた今の状態
  ではなく、道床は1個の均一な高さのまま、手前・奥のレールを3D的に手前側へ寄せる
  ことでレンダリング後に画面上3px手前に描画されるようにしてほしい、道床の奥側が
  相対的に広くなるはず、との提案を受ける
- 対応: 道床を単一boxに戻し（手前の高さのみ、分割しない）、レール側をworld X方向に
  手前(+X)へ寄せることで画面縦方向の下げを実現する方式に変更。world X 1BUの
  オフセットは画面(dx=+64,dy=-32)/BUを動かすため、画面縦方向にn_px下げたい場合は
  `ΔX = n_px / 32`を+X方向に足せばよい（水平方向の投影量64ではなく、この式では
  画面縦方向の投影量32を使う点に注意）。手前レールは`+1/32`BU、奥レールは`+3/32`BU
  をそれぞれの元のゲージオフセットに追加。結果としてレールのX位置がゲージ中心から
  ずれ、道床(位置不変)に対して奥側の余白が相対的に広くなる
- 再レンダリング→postprocess→pak再ビルド→addonsフォルダへ再配置。道床の段差は
  解消され、レール間の画面上のゲージ幅は10pxより狭くなった（副作用として許容）
- ユーザーによる再確認: 「手前・奥の線路両方が手前に寄りすぎている。横に4px分ぐらい」
  と指摘。計算上は手前+2px、奥+6px横方向にずれており、平均するとおよそ4pxで指摘と一致
- X/Y/Zそれぞれの副作用（Z=道床への埋没/浮遊、Y=継ぎ目破壊、X=ゲージのズレ）を整理して
  ユーザーに提示し、方針を確認。「Xのズレを少し減らし、残りをZ(道床の小さな段差)で補う」
  という折衷案を選択
- 対応: 必要な2px差（手前1px・奥3px）を、道床のごく小さな段差(1px)とレールのXシフト
  (1px)に半分ずつ分割する実装に変更。道床を手前/奥の2つのboxに分割するが、段差は
  以前(2px相当)の半分(1px)に抑える。奥レールのXシフトも`FAR_EXTRA_DOWN_PX=3`全体分
  ではなく`FAR_RAIL_X_SHIFT_PX=1`分だけに縮小。結果、ゲージのズレは奥+2px程度
  （以前の+6pxから縮小）、道床の段差も1px（以前の2px相当から縮小）に収まる
- 再レンダリング→postprocess→pak再ビルド→addonsフォルダへ再配置

## 得られた知見や失敗
- `obj=way` は building と違い、方向ごとの画像キー(`Image[n]`, `Image[ns]`, `Image[nsew]`...)を
  大量に要求する構造だが、makeobjは `image[-][0]` さえあればfatalにならないため、PoC段階では
  必要最小限のキーだけ埋めて残りを省略できる（未対応方向は無表示になるだけで、ビルド自体は通る）
- `icon=> filename.row.col` の先頭 `>` は「ズーム時に拡大縮小しない」フラグ
  （`image_writer.cc:356-360`）。building側のdatには存在しない書式で、way系の実データを
  読んで初めて気づいた。今回のdatにも踏襲したが、動作上必須ではない（付けなくてもbuildは通る）
- 高さを「1フロア相当に対する割合×isometric補正係数」で計算する方式にしたことで、
  立体物(box, 割合=1.0)と平面に近い物(box, 割合0.1前後)を同じ考え方で扱えることを確認。
  次に立体物と平面物を隣接させて高さの見た目の連続性を検証する余地がある
- **カメラ注視点Zは「オブジェクトの高さ」ではなく固定の地面アンカー(0.408)**。
  これまでの立体物テスト(isometric_box/station_cube)はどちらも高さ0.816のcubeで
  「オブジェクトの中心」と「固定アンカー0.408」がたまたま一致していたため、
  この2つの違いが表面化していなかった。平面(高さ≒0)を扱って初めて誤りが露呈した。
  以後、高さの異なる形状を扱うすべてのレンダリングスクリプトで
  `target.z = 0.408`固定を徹底する必要がある（knowledge.mdに追記済み）
- **細い/薄い形状の寸法はBU単位の相対値ではなくpx単位で直接指定する方が実用的**。
  「1フロア相当に対する割合」という考え方は立体物同士の比較には便利だが、
  ユーザーが実物のpak128画像と見比べて「あと1pxくらい」とフィードバックする場合、
  px単位の変換式を用意しておいた方がフィードバックをそのまま反映できる
- **カメラのright/up基底ベクトルを実測すればpx⇔BU変換係数を厳密に求められる**。
  `cam.matrix_world.to_3x3()`で得たright/upベクトルと世界軸の内積 × `(解像度/ortho_scale)`
  で、世界のどの軸方向のオフセットが画面上で何pxになるかを正確に計算できる
  （今回のカメラ設定では垂直(Z軸)は`(128/√2)×cos30°≈78.38px/BU`、水平(X軸・Y軸)は
  `(128/√2)×cos45°=128/2=64.0px/BU`とちょうど半分になるきれいな値だった）。
  この値はknowledge.mdに追記済みで、他のwayバリエーション(カーブ・分岐等)を
  作る際にも同じ変換式を使い回せる
- **同じ1BUの世界オフセットでも、X軸方向の位置ズレはZ軸(高さ)方向にも画面上の
  ズレとして現れる**。今回の2本のレールはZ座標(高さ)は本来同じ値で作っていたが、
  X座標(横位置)が異なるため画面上の縦位置が自然にズレる。さらに実機で見ると
  手前・奥のレールで必要な追加補正量(1px vs 3px)も異なっていた。細かい形状を
  複数個並べるときは、たとえ元のZ座標を揃えていても画面上の見え方は個別に
  検証・補正する必要があることが分かった
- **画面上の見た目のズレ補正はZ(高さ)だけでなくX/Yの組み合わせでも実現できる。
  細い形状で高さの余裕が乏しいときはX/Y側で補正する方が安全**。
  Z(高さ)を直接減らして画面上の位置を下げる方法は直感的だが、対象が薄い形状の場合
  すぐに他のオブジェクト(道床)の内部に潜り込んで不透過な面の陰に隠れ、消失する
  リスクがある。world X/Yを逆符号で組み合わせれば、Zを一切変えずに画面縦方向だけの
  移動を作れるため、こちらの方が安全に「見えた」（knowledge.mdに変換式を追記済み）
- **ただし上記のX/Y組み合わせ技は「タイル境界と厳密に一致すべき方向」には使えない**。
  レールのように長さ方向(Y)がタイル境界ぴったりに設計されている形状にYオフセットを
  加えると、画面上の見た目は正しくても実際のワールド座標上のfootprintがタイル境界から
  ずれ、隣接タイルとの継ぎ目に隙間が生じる。見た目の位置合わせ(screen dx/dy)と
  タイル境界への整合(world座標そのもの)は別の要求であり、後者を壊さない軸
  (この場合はZのみ)に補正手段が限定されることがある
- **「画面上のズレを直したい」だけの場合でも、対象の他の制約（タイル境界と厳密に
  一致すべき軸はどれか、他オブジェクトとの高さの余裕はどれだけか）を先に確認してから
  補正方法(X/Y/Zのどれを使うか)を選ぶべき**。今回は「Yは動かせない(継ぎ目)」
  「Zは1pxまでしか動かせない(道床への埋没)」「Xを動かすとゲージが変わる(承認済み値)」
  という3つの制約が同時に成立し、要求された「手前1px・奥3px」を幾何学的に両立
  できないことが判明。この場合はコードで無理に帳尻を合わせず、ユーザーに
  トレードオフを提示して判断を仰ぐのが適切だった
- **タイルの外周とちょうど一致する形状は、アンチエイリアスによる半透明エッジがそのまま
  タイル継ぎ目の隙間として見えてしまう**。厚み方向の話（今までの知見）とは別の軸の問題で、
  平面(道路)の輪郭がタイル境界と一致する場合に固有。4倍supersample+ダウンサンプル後に
  アルファを二値化する対策で解消できた。この手法は境界がタイル境界と一致する他の
  way形状（線路の直線区間なども厳密にはタイル境界に接する）にも同様に適用が必要
- **半透明を無くしても境界の色みの差は別問題として残りうる。原因はRGBの黒背景混入
  だけでなく、Pillowの`Image.resize()`がRGBA画像に対して行うアルファ考慮の
  内部処理そのものにもある**。透明領域のRGBを塗りつぶす(color bleed)だけでは
  不十分で、RGBAのまま一括でresizeする限りPillow内部の処理でフリンジが再発した。
  RGB(3ch)とアルファ(1ch, "L"モード)を完全に別画像としてresizeし、最後にマージする
  ことで初めて解消した。「半透明が無くなった」ことと「色が均一」であることは別の
  検証項目であり、両方を実際のピクセル値で確認する必要があった
  （knowledge.mdに追記済み）
- **このパイプラインでは「3Dモデルとしての物理的な正しさ」より
  「レンダリング後の2Dスプライトとしての見た目の正確さ」を優先してよい**。
  Blenderは最終的にpak128の2Dスプライトを作るための道具であり、3Dシーン自体が
  ゲーム内で使われるわけではない。オブジェクトが地面より下に沈む・複数オブジェクトの
  Z関係が現実の物理と矛盾する、といった3D的な不自然さがあっても、2D出力の見た目が
  正しければ問題ない。今回「道床を地面より下に沈める」という3D的にはおかしい配置で
  レールとの段差を復元したのはこの方針に基づく（ユーザーからの明示的な方針表明）
- **同じ「奥に3px下げる」操作でも、周囲の他オブジェクトの配置次第で結果(隠れる/隠れない)
  が変わる**。道床のZ範囲が`[0,1px]`だったときは奥のレールが道床の内側に潜り込んで
  隠れたが、道床を`[-1px,0px]`に下げた後は同じ3pxの下げが道床の範囲と重ならず、
  隠れずに描画された。「この量のシフトは安全か」は固定の答えではなく、その時点の
  周辺オブジェクトの配置に依存するため、寸法を変更するたびに再検証が必要
- pak128実物のような、道床全面を覆う枕木+細いレールという構造は、今回の
  「幅を絞った道床+2本の細い棒」という簡略モデルとは大きく異なる。デフォルト
  アセットとの完全な互換（レール位置・道床幅の完全一致）を目指すなら実物画像の
  精密なピクセル測定が必要で、これは平面/薄い形状のPoCという当初のスコープを
  超える作業になる（別タスクとして扱うのが妥当）
- **「隠れて消える」と「浮いて/ぶら下がって見える」は原因も対処も別物で、区別するには
  実際のレンダリング画像のピクセル座標を見るしかない**。ユーザー報告の「消えた」を
  文字通り受け取り「非表示になる不具合」として調査したが、実際はデータとしては
  存在しつつ「道床から切り離れて浮いた/ぶら下がった」見た目の問題だった。
  抽象的な報告と実際の画像診断の間にズレがあり得るため、可能な限りraw画像の
  ピクセル値を直接確認して判断するのが確実
- **「オブジェクトAをオブジェクトBから突き出て見えるようにする」ためには、Aを動かす
  だけでなくBの対応する部分も一緒に動かす必要がある場合がある**。奥のレールだけを
  下げても、支える側の道床が追従していなければ「浮いた/ぶら下がった」見た目になる。
  今回は道床を2分割して個別に沈めることで解決したが、一般に「複数パーツが積み重なる
  形状」で一部だけ位置調整する際は、隣接パーツとの接続関係も一緒に見直す必要がある
- **画面上の位置ズレ補正には最低3通りの軸選択肢(X/Y/Z)があり、それぞれ副作用が異なる。
  「何を変えずに済ませたいか」に応じて選ぶ軸を変える必要がある**。今回の一連の試行錯誤
  をまとめると: Z(高さ)は他オブジェクトへの埋没/浮遊リスク、Y(進行方向)はタイル
  継ぎ目破壊のリスク、X(横方向)はゲージ(見た目の幅)が変わるリスクを持つ。
  「継ぎ目を壊さない」「道床の形状を保つ」等、優先したい制約が変わるたびに、
  適切な軸の組み合わせも変わる。1つの正解があるわけではなく、その時点で
  何を犠牲にできるかをユーザーに確認しながら選ぶ必要があった
- **X/Y/Zいずれの軸を使っても副作用をゼロにはできない場合、複数の軸に按分する
  折衷案が有効**。1つの軸だけで解決しようとすると副作用が全部その軸に集中して
  目立つが、複数軸（今回はZの道床段差とXのゲージズレ）に半分ずつ分けることで、
  それぞれの副作用の大きさを目立たない程度まで縮小できる
- ユーザーによる最終確認: 「OKそろった」。あわせて「結構調整しないとそろわないね。
  8方位揃えようとすると大変そう」とコメントを受け、この単一方向の調整だけで
  12周のフィードバックと試行錯誤が必要だったことを踏まえ、8方位すべてを個別に
  作り込むのはPoCのスコープ外として区切り、ここで完了とすることに合意
- **1方向の位置・寸法をpak128実物と合わせ込むだけでも、表示位置・太さ・タイル境界の
  半透明化・色みのフリンジ・進行方向の継ぎ目・段差・デフォルトアセットとの相互接続、
  と非常に多くの検証軸がある**。building(駅拡張建物)のような孤立した立体物では
  問題にならなかった「タイル境界に接する」「複数方向で見た目が変わる」という
  way特有の制約が、平面/薄い形状を扱って初めて次々と表面化した。8方位・カーブ・
  分岐まで含めた本格的なway実装は、この検証コストを8倍(以上)にする覚悟が必要で、
  roadmap.mdの中期TODO「複数タイルサイズ対応」やvehicleの「方向マッピングの
  キャリブレーション工程をスクリプト化」と同様、手作業の反復を減らす仕組み
  （例: 一方向の調整結果から他方向を自動導出する、等）を検討する価値がある
