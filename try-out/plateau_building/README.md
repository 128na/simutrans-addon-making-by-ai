# PLATEAU 3D都市モデルの建物をSimutransアドオン化

作業日: 2026-07-07

## 目標
国交省PLATEAUの3D都市モデルから、実在建物をSimutransのイソメトリック建物アドオンに変換できるか検証する。
- step1: 建物1棟を手作業ベースで変換してみる
- step2: 変換フローをPythonで自動化できるか検証する
- step3: step1をゲーム内表示した際に見つかった「表示位置の北東寄り」「スケールが小さい」を修正する
- step4: step3後もゲーム内で残っていた接地位置のズレ（接地アンカー規約の取り違え）を修正する
- step5: モデルデータから「窓など発光色に置き換える場所を区別できるか」「実物スケールを
  認識できる情報があるか」を調査する（今後の自動化・夜間照明表現に向けた前提調査）
- step6: BackImage[0]〜[3]に異なる画像を割り当てる「四方位対応」を試す（これまでは
  station_test由来のDims=1,1,4を流用しつつ4スロットとも同一画像だった）

## 結果
達成。1棟変換・pak化・自動化スクリプト・スケール修正・接地アンカー修正のすべてを
ゲーム内表示で最終確認済み（ユーザーより「狙った通りの位置に表示できている」と確認）。
step5の調査も完了（結論は「試したこと」参照。窓は幾何/意味情報としては区別できず、
スケールはOBJのヘッダコメントから確実に確認できる）。
step6も実機キャリブレーション完了。「建設ダイアログ表示」と「線路隣接設置時の実際の
向き」で対応関係が1つズレる（`placement(l)=dialog(l-1)`）という知見が得られたが、
4方向自体は矛盾なく順序通り回転しているため、今回のdatの割り当て修正は不要と判断
（詳細は「試したこと」参照）。

## 試したこと

### データ入手
- PLATEAUはCityGML形式が基本だが、G空間情報センターが東京都23区等の主要都市について
  OBJ/FBX変換済みデータをS3で配布している
  （[plateau-tokyo23ku](https://www.geospatial.jp/ckan/dataset/plateau-tokyo23ku)）
- 東京23区全体のOBJ zipは約2.1GBだが、`remotezip` (pip) でHTTP Range Requestを使い、
  zipをダウンロードせず必要な建物1棟分（obj+mtl+texture、数十KB〜）だけを直接取得できた
- LOD1（直方体押し出しのみ）とLOD2（屋根形状・テクスチャあり）が混在。LOD2は23区データ中
  80棟のみ（ランドマーク級の建物が中心）
- 採用: メッシュコード `53394515`（LOD2、フットプリント 58.8m×45.6m、高さ22.0m、
  テクスチャ1枚）

### Blenderインポート
- `bpy.ops.wm.obj_import(forward_axis='Y', up_axis='Z')` でそのままインポート可能
  （PLATEAU公式チュートリアルの「前方の軸=Y、上向きの軸=Z」に合致。CityGML由来の座標系は
  すでにZ-upのため軸変換は不要）
- mtlの相対パスがWindows形式(`.\materials_textures\xxx.jpg`)だったが、Blender 5.1の
  OBJインポータはWindows環境でそのまま解決できた
- 実寸メートル単位のメッシュはそのまま既存のイソメトリックカメラ設定
  （仰角60°/方位45°, ORTHO）に投入できる。station_test で使った「1BUキューブに
  0.816のZスケールを掛ける」処理は不要（あれは単位キューブを見た目上ただしく見せるための
  補正であって、実寸ジオメトリには適用しない）

### タイル内スケール決定
- Simutrans 1タイル ≒ 125m という想定値を置き、`ortho_scale = 125 × √2` で
  カメラを設定 → 実寸フットプリントがタイル内に収まるかを機械的に判定できる
  （`max(footprint_x, footprint_y) <= 125` で1×1タイル建物として成立するか判定）

### dat作成・pak化
- building type はまだ`res`/`com`/`ind`（市街地建物）を検証しておらず、station_test で
  動作確認済みの `type=extension` + `waytype=track` パターンを流用した
- `dat_linter lint` → `makeobj pak128` の順で問題なくビルドできた
- makeobjは`building.png`を自動クロップ（128×128 → 実質46×26相当）。今回は4方向とも
  同一画像のため方向間のクロップ差異問題（vehicle検証で見つかったもの）は発生しない

### 自動化スクリプト（[convert.py](convert.py)）
- 「zip URL + zip内の建物ディレクトリパス + 出力名」を渡すだけで、
  ダウンロード→フットプリント計算→Blenderレンダリング→アイコン合成→dat生成→lint→pak化
  を無人実行できることを確認
- 別のLOD2建物（メッシュコード `53393664`）でも通しで動作した

### 表示位置・スケール修正(step3)

- ゲーム内表示確認で「表示位置が北東寄り」「スケールが小さい」の2点が見つかった
- `refs/simutrans/src/simutrans/descriptor/writer/image_writer.cc` を読み、原因を特定:
  - `write_obj()` が画像を読み込む際に `init_dim()` でPNGの不透明ピクセルからバウンディング
    ボックスを自動検出し、`image.x = dim.xmin`, `image.y = dim.ymin` としてそのままpakに
    書き込む。`obj/gebaeude.cc` の `display()` は `display_normal(image, xpos, ypos, ...)` で
    描画しており、この `image.x/y` がタイル基準点からの実際の描画オフセットになる
  - `TILE_SIDE_M=125` 固定値は実際の建物フットプリント(58.8m×45.6m)よりずっと大きく、
    レンダー結果は128×128キャンバスの中央にごく小さく(クロップ後46×26px)写っていた。
    クロップ位置が理論上のキャンバス中央からわずか数px(実測2〜6px)ズレていても、
    描画サイズが小さいほどそのズレが相対的に大きく見える、というのが「北東寄り」の
    実体だったと判断した（画像自体の縦横中心はほぼキャンバス中央に一致しており、
    アンカー計算自体が壊れていたわけではない）
  - `knowledge.md`「vehicleの接地位置ズレ対策」で既に有効性が確認されている
    「4隅への不透明1pxマーカー焼き込みで128×128フルキャンバス化」が、この
    クロップ由来のズレを構造的に排除する対策としてbuildingにもそのまま適用できると判断
- 修正内容（`import_and_render.py` と `convert.py` のBlenderテンプレート両方に同じ内容を適用）:
  - スケール: `TILE_SIDE_M`固定値をやめ、`max(span_x, span_y, span_z) * √2 * 1.05`
    （フットプリント/高さの最大辺を基準に5%マージンでタイルに自動フィット）に変更
  - 位置: レンダー後に`bpy.data.images`でPNGを読み直し、4隅(`(0,0)`,`(127,0)`,`(0,127)`,
    `(127,127)`)に不透明(alpha=1.0)の黒1pxマーカーを焼き込んでから保存
  - `import_and_render_zoom.py`（見た目確認用の使い捨てスクリプト）は自動フィットが
    本採用になったため削除し、`building_zoomed.png`も削除
  - `convert.py`冒頭の未使用だった`BLENDER_TEMPLATE`定数、および`--tile-side-m`引数と
    それに依存していた「1タイルに収まるか」判定メッセージを削除（固定タイル実寸想定を
    撤廃したため）
- 検証: `makeobj VERBOSE DEBUG pak128 plateau_building.pak plateau_building.dat` の
  image debug出力（列は `X Y OffX OffY Width Height`）が
  修正前 `0 0 42 54 46 26` → 修正後 `0 0 0 0 128 128` に変化し、4レイアウト全てで
  オフセット差異が構造的になくなったことを確認。`dat_linter lint`も引き続き通過。
  `convert.py`も既存の成功済み建物(メッシュコード`53394515`)で再実行し、同一の
  `0 0 0 0 128 128`が出ることを確認した（`auto_53394515_v2/`に生成、検証用途で残置）
- クロップ前の実シルエット(マーカーを除く可視ピクセル)のbboxは `x:21-111, y:44-95`
  （128×128キャンバス中心は63.5,63.5）で、シルエット中心は(66, 69.5)とほぼ中央に一致。
  修正前の46×26pxに比べ90×51pxまで拡大し、「タイル目一杯」の見た目に近づいた
- ゲーム内での実機確認（xposへの実際の描画結果）はこのtry-outの範囲では未実施
  （ユーザーが後日確認予定）

### 接地アンカー位置の追加修正(step4)

- step3修正後にユーザーが実機確認したところ、スケールは適正化されたが表示位置は
  依然として北東寄りのままだった、というフィードバックを受けた
- ユーザー側で「128×128のpak画像は下半分が地面(ground)扱いで、実際のベースタイル
  （接地面）は画像下端起点で `(64,0),(128,32),(64,64),(0,32)` を結ぶ菱形であり、
  その中心（画像下端から32px = 上端起点でrow96）が正しい接地アンカー」という
  決定的な情報が得られた
- 根本原因: step3の実装は `target_z = span_z / 2`（建物自身の高さの中心）を
  キャンバス中心(row64)に合わせていたが、これは一般化できていなかった。
  `station_test`（1BUキューブ、高さ0.816・中心z=0.408）がたまたま
  `0.408 × cos30° × (128/√2) ≈ 32px` のオフセットとなり正しい接地点(row96)と
  一致していたため見た目上問題が出なかっただけで、フットプリント/高さ比が
  station_testの1BUキューブと異なる実建物（PLATEAU建物）には成り立たない仮定だった
- 修正内容（`import_and_render.py` と `convert.py` のBlenderテンプレート両方に同じ内容を適用）:
  - `target_z = span_z / 2` を廃止し、`shinkansen_0/render_0series.py`で実証済みの
    変換式 `PX_PER_UNIT = 128 / ortho_scale × cos(30°)` を流用した
    `target_z = 32 / (128 / ortho_scale × cos(30°))` に置き換え。
    建物自身の高さ(`span_z`)には依存せず、`ortho_scale`（＝タイルフィット計算結果）
    のみから接地アンカー位置を算出するようにした
- 検証: レンダー画像の可視シルエットbbox（4隅マーカーを除く）が
  修正前 `y:44-95`（中心69.5）→ 修正後 `y:62-113`（中心87.5）に変化し、
  約18px下（画像下端方向）にシフトした。これは理論値
  （`target_z`の変化量14.2m × `px_per_unit`(≈1.269px/m) ≈ 18.0px）とほぼ一致しており、
  式の実装が意図通り機能していることを確認した
  - `makeobj VERBOSE DEBUG` のOff X/Y/Width/Heightは4隅マーカー焼き込みにより
    引き続き `0 0 128 128`（クロップ無効化は維持されたまま）
  - `dat_linter lint`は引き続き通過
  - `convert.py`も既存の成功済み建物(メッシュコード`53394515`)で再実行し、
    `import_and_render.py`が生成した`building.png`とピクセル単位で完全一致する
    画像が生成されることを確認した（`auto_53394515_v2/`に生成）
- ゲーム内での最終確認: ユーザーが実機ロードし「狙った通りの位置に表示できている」ことを確認済み

### モデルデータの調査(step5): 窓の識別・実物スケールの認識

今後、夜間に窓を発光色へ置き換える表現や、複数建物の相対スケールを自動決定する際の
前提となる、モデルデータ自体が持つ情報を調査した。OBJではなく、より情報量の多い
元データ(CityGML)を対象に確認した（G空間情報センターの同一データセットにCityGML zipも
配布されている: `13100_tokyo23-ku_2020_citygml_3_2_op.zip`）。53394515の建物を含む
1kmメッシュのCityGML(`53394515_bldg_6697_2_op.gml`, 約19MB, 建物2143棟収録)を
`remotezip`でダウンロードし、XMLを直接grepして確認した。

**Q1. 窓など発光色に置き換える場所は区別できるか → できない（LOD2の範囲では）**
- このメッシュ内には`bldg:WallSurface`(51件)・`bldg:RoofSurface`(8件)・
  `bldg:GroundSurface`(1件)のみが存在し、`bldg:Window`/`bldg:Door`/`bldg:opening`は
  **1件もヒットしなかった**（`grep -c`で0件）
- CityGMLの仕様上、窓・ドアの開口部を独立した意味情報付きジオメトリとして持てるのは
  LOD3以降。このメッシュには`lod3Solid`/`lod3MultiSurface`も0件で、そもそもLOD3データが
  存在しない（LOD3はPLATEAU全体でもごく一部のランドマーク建物のみに限られる）
- 実際にwallの外観テクスチャ(`skjp5921.jpg`、OBJ変換後も同名で同梱されている)を見ると、
  窓らしき小さな暗い矩形パターンやガラス面のグリッド状の格子模様は**画像として**
  はっきり写っている。つまり「窓の見た目」自体はテクスチャ写真の中に含まれているが、
  それが「窓である」という意味づけ・独立したUV領域や材質分けは一切されておらず、
  1枚の外壁写真に焼き込まれているだけ
- 結論: LOD2までしか無い（Tokyo23区データの大半がこれに該当）建物では、モデルの構造化
  データ(ジオメトリ・マテリアル名・意味タグ)から窓位置を機械的に特定する方法は無い。
  やるとすれば、テクスチャ画像そのものに対する画像処理（暗い矩形の格子パターン検出、
  明るさ/色によるセグメンテーション、あるいは物体検出モデル）が必要になる

**Q2. 実物スケールを認識できる情報はあるか → OBJ側にある。CityGML側は要注意**
- 配布されているOBJファイルはヘッダコメントに変換元CRSを明記している
  （`# COORDINATE_SYSTEM: OGC_DEF PROJCS[...,UNIT["metre",1,...],AUTHORITY["EPSG","6677"]]`)。
  EPSG:6677は「JGD2011 / Japan Plane Rectangular CS IX」という平面直角座標系で、
  単位は明示的に`metre`。つまりOBJの頂点座標は**そのままメートル単位**であることが
  ヘッダを1行パースするだけで機械的に確認できる（このtry-outでは実際にこの前提で
  フットプリント計算・タイルフィットを行い、正しく機能した）
- 一方、**元のCityGMLの座標は緯度経度(度)**である点に注意が必要。今回確認した
  ファイルは`srsName="http://www.opengis.net/def/crs/EPSG/0/6697"`
  （JGD2011の水平成分はEPSG:6668 = 緯度経度度単位、高さ成分のみメートル）で、
  実際の`gml:posList`の値も`35.68... 139.68... 58.836`（緯度, 経度, 標高m）という
  度単位の並びだった。CityGMLを直接扱う場合は、この`srsName`を見ずに座標をそのまま
  メートルとして扱うと1000倍以上スケールを誤る。今回配布されているOBJ/FBXは
  FME等の変換ツールが平面直角座標系(メートル)へ**あらかじめ再投影した上で**
  ヘッダにその旨を明記してくれているため、Blenderパイプラインではこの恩恵を
  受けられていた、という位置づけになる

### 四方位対応の検証(step6)

- これまでの`plateau_building.dat`は`Dims=1,1,4`（station_test由来のパターン流用）
  でありながら、`BackImage[0]`〜`[3]`すべてに同一の`building.png`を割り当てており、
  実際に4方向で異なる絵を出す検証はしていなかった。今回、基準建物(53394515)を
  Z軸回転させた4枚の画像を実際にBackImage[0]〜[3]へ割り当てて試した
- [render_4dir.py](render_4dir.py): `import_and_render.py`と同じカメラ・接地アンカー
  ロジック（フットプリント自動フィット + 4隅マーカーによるクロップ位置ズレ対策）を
  流用し、Z軸回転角(0°/90°/180°/270°)を引数で受け取れるようにしただけの変更。
  回転後にフットプリント(span_x/span_y)を再計算しているため、非正方形の建物でも
  各回転で正しくフィットする
- [bake_label.py](bake_label.py): shinkansen_0のvehicle方向較正
  （Z回転角ラベルを画像に焼き込んで実機で直読みする方式）と同じ考え方を建物にも
  適用。ただしvehicleは`emptyimage[方向名]`という名前付きキーだったのに対し、
  buildingの`BackImage[l]`は`l=0,1,2,3`という**数値インデックス**なので、
  焼き込むラベルも「回転角」ではなく「スロット番号(L0〜L3)」にした
  （l=0→0°, l=1→90°, l=2→180°, l=3→270°という素朴な対応を仮定して割り当てただけで、
  この対応が実際のマップ方位と一致しているかは未検証）
- `dat_linter lint`・`makeobj`ビルドとも成功。4画像とも4隅マーカーにより
  128×128フルキャンバスを維持（クロップ起因の位置ズレは起きない）
- **実機キャリブレーション結果**: ユーザーが実機で確認したところ、**2つの異なる
  文脈で異なる対応関係**になっていることが判明した
  - 方角指定で建設するダイアログ表示: `L0`=南, `L1`=東, `L2`=北, `L3`=西
  - 実際に線路（ホーム）に隣接する形で設置したとき: `L0`=西隣り, `L1`=南隣り,
    `L2`=東隣り, `L3`=北隣り
  - 両者を突き合わせると、「設置時」の対応は「ダイアログ」の対応を**l方向に1つ
    ずらしたもの**になっている（`placement(l) = dialog(l-1 mod 4)`。例:
    `dialog`ではl=0が南だが、`placement`ではl=0は西＝`dialog`のl=3の方角）。
    回転の向き自体（l増加につれてS→E→N→Wと反時計回りに90°ずつ進む）は両者で共通
  - 今回の4枚は「l=0→Z回転0°, l=1→90°, l=2→180°, l=3→270°」という単純な規則で
    生成しており、どの文脈で見ても4方向が矛盾なく順序通りに回転して見える
    （実際に上記の対応関係が一貫した90°刻みになっていることからも確認できる）。
    そのため**このdatの割り当てを修正する必要はなく**、この「ダイアログ表示と
    設置時の実際の向きは1つズレる」という関係性自体が、`type=extension`
    （station拡張建物）を今後作る際に踏まえておくべき知見となる

## 得られた知見や失敗
- **OBJファイル1つ=建物1棟とは限らない**: メッシュコード`53393664`のOBJは
  ファイルサイズこそ小さい(45KB)が、フットプリントが265.6m×162.7mと1タイルに収まらず、
  レンダー画像を見ると実際には複数の独立した構造物（`o`/`g`によるグループ分けなし）が
  1ファイルに混在していた。ファイルサイズや単一ファイルという見た目だけで
  「1棟の建物」と判断するのは危険で、フットプリントの実測とレンダー結果の目視確認が必須
- **メートル単位の実寸モデルは0.816のZスケール補正が不要**: この補正は「1BUの
  単位キューブ」を見た目上正しい高さに見せるためのものであり、実寸ジオメトリ
  （PLATEAUモデル含む）にそのまま適用すると誤り。既存の車両検証知見と合わせ、
  「補正が必要なのは抽象化されたプリミティブだけ」という前提を明確化できた
- **タイル内フットプリント判定はPythonだけで完結する**: Blenderを起動せずOBJの
  頂点座標をテキストパースするだけでフットプリントは計算できるため、
  「1×1タイルに収まる建物だけを機械的に選別する」前処理が現実的
  （Blender起動はレンダリング工程だけで良い）
- **remotezipでzip全体ダウンロードを回避できる**: S3ホスティングのzipはHTTP Range
  Requestに対応しており、`remotezip`ライブラリで特定エントリだけをストリーム取得できる。
  数GBのzipから数十KBの建物データだけを抜き出す用途に有効
- **小さくクロップされた画像ほど数pxのアンカーずれが目立つ**: makeobjは不透明ピクセルの
  バウンディングボックスだけを保存し、そのオフセットがそのままゲーム内の描画位置になる
  （`image_writer.cc`の`init_dim`/`write_obj`）。フットプリントがタイルよりずっと小さい
  状態でレンダーすると、クロップ後の画像は数十pxしかなく、理論上センターからのわずか
  数pxのズレでも見た目の比率としては大きく見える。station_test（1×1BUの箱がタイルを
  ほぼ埋め尽くす）で問題が顕在化しなかったのはこのため。「表示スケールを適正化する」こと
  自体が「見た目の位置ズレ」も同時に緩和する、という関係にあると分かった
  （※ただしこれは位置ズレの一因に過ぎず、真因は下記の接地アンカー規約の取り違えだった。
  スケール修正だけでは北東寄りは解消しなかったと後日の実機確認で判明している）
- **pak128の128×128タイル画像は「接地アンカー=画像下端から32px(row96)」が正しい規約**:
  画像の下半分が地面(ground)扱いで、ベースタイルの菱形
  `(64,0),(128,32),(64,64),(0,32)`（画像下端起点）の中心が接地点になる。
  カメラターゲットのZは「建物自身の高さの中心(`span_z/2`)」のような対象物依存の値ではなく、
  「`32 / (128 / ortho_scale × cos30°)`」という`ortho_scale`のみに依存する値にする必要がある。
  `station_test`の1BUキューブでは対象物依存の値(`span_z/2`)がたまたまこの正しい値と
  一致していたため、汎用性の低い実装のまま「動いているように見えて」いた。
  1つの検証ケースで正しく見えても、それが偶然の一致でないかを疑う必要がある教訓
- **LOD2までの窓は「写真として写っているだけ」で意味情報を持たない**: PLATEAUの
  `bldg:Window`/`bldg:opening`はLOD3以降でしか出現せず、Tokyo23区データはLOD3を
  1件も含まない。夜間発光窓のような表現をやるなら、モデルデータでなくテクスチャ画像への
  画像処理（パターン検出等）が必要という前提を早い段階で確認できた
- **実物スケールの根拠はOBJヘッダのCRS宣言、CityGML生座標は緯度経度(度)**:
  配布OBJの`# COORDINATE_SYSTEM`コメントに`UNIT["metre"...`とあることをもって
  「頂点=メートル」を機械的に確認できる。これはFME変換時に平面直角座標系へ
  再投影された結果であり、CityGMLの生の`gml:posList`は緯度経度(度)なので、
  CityGMLを直接パースする場合は再投影(pyproj等)が必須になる
