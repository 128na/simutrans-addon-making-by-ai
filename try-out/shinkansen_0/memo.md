# 0系新幹線 先頭車両 (Phase 1)

## 目標
車両（vehicle）タイプのアドオンをBlender→8方向レンダリング→makeobjのパイプラインで
実用的に生成できるか検証する。0系新幹線先頭車両を題材に、簡易形状（直方体+ノーズ球+帯）で
ゲーム内に正しい向きで走行表示させることを目指した。

## 結果
達成（方向・接地位置・流線型シルエットは及第点。0系固有の特徴表現は次フェーズ課題として区切り）

## 試したこと
- Blenderで車体(直方体)+ノーズ(楕円体)+青帯+台車の簡易モデルを親エンプティに紐付けて作成
- 親エンプティをZ軸45°刻みで回転させ8方向(0°/45°/.../315°)を128×128でレンダリング
- vehicle_writer.cc を読み、vehicle dat のキーが `emptyimage[方向]` (方向: s,w,sw,se,n,e,ne,nw) であると確認
- `freight=passenger` はpak128に存在せず `freight=Passagiere`（ドイツ語）が正しいgood名と判明
- 方向マッピングを机上の三角関数で導出 → 2回とも実機で不一致（向きがズレる/マッチ棒状に見える）
- 数式的推測を諦め、各レンダーにZ回転角ラベル(`Z0`〜`Z315`)をPillowで焼き込んだキャリブレーション用dat
  (`shinkansen_0_calib`)を作り、実機で8方向走行させてラベルを直接読み取る方式に切替
- 実測データから「N/Sはそのまま正しいが、E/W・NE/NW・SE/SWがペアで入れ替わる(東西軸の鏡映)」
  という規則性を発見し、dat修正 → 全方位正しく表示されることを確認
- 方向は直ったが「線路に浮いて見える」「車体がマッチ棒/カプセル状」の指摘を受け追加対応：
  - makeobjの自動クロップが回転ごとに異なるバウンディングボックスを検出し、オフセットが
    レンダーごとにバラついていた疑いがあったため、全8枚の4隅に不透明1pxマーカーを焼き込み
    強制的に128×128フルキャンバスでパックさせるよう変更（クロップ起因のズレを構造的に排除）
  - 「20px下にずらすと線路に接する」という実測フィードバックを受け、カメラターゲットZを
    `20px / (128/ortho_scale * cos30°)` 分だけ計算してオフセットし解消
  - 球(UV Sphere)をスケールしただけのノーズは断面が常に丸く涙滴形にならないと判明 → 円錐に
    変更したら今度は直線テーパーで尖りすぎ(ピラミッド状) → 最終的にbmeshでリング(輪切り)を
    積み上げる自作ロフト関数に変更。半径を`sqrt(1-t^2)`カーブで減衰させ先端付近まで太さを
    保つことで「団子鼻」形状を再現
  - 車体・ノーズ・青帯を別オブジェクトで作ると「□〇」(箱と球が分離して見える)や帯がテーパーに
    追従しない問題が発生 → 車体とノーズを同一bmeshメッシュ(角丸長方形断面のスーパー楕円)に
    統合し、青帯も別オブジェクトでなくZ位置に応じたマテリアルインデックス割り当てに変更。
    これにより継ぎ目・帯のミスマッチが構造的に解消

## 得られた知見や失敗
- vehicle dat の画像キーは `emptyimage[方向]`（建物の`BackImage[l][y][x][h][p]`とは別書式）
- 方向コードは `s, w, sw, se, n, e, ne, nw`（vehicle_writer.cc の `dir_codes` 配列順）
- pak128の旅客good名は `Passagiere`。`freight=passenger`は `Cannot resolve 'GOOD-passenger'` で
  起動時FATAL ERRORになる（addons配置前にgood名を確認すべき）
- カメラ・回転・isometric投影から方向対応を三角関数で導出するのは罠が多く、実機キャリブレーションの
  方が確実だった。具体的には「Blenderの回転方向(CCW/CW)とコンパスラベルの対応」「ゲームエンジンが
  dat の方向キーをどう内部方向(ribi)に対応付けるか」の両方に誤った仮定を置きやすい
- 最終的な正しい対応は「N/S不変、E/W・NE/NW・SE/SWが入れ替わる」という東西軸の鏡映だった
  （原因はBlenderモデルかカメラのどちらかでX軸の向きについて暗黙の仮定がズレていたためと推測、
  未特定。次回モデルでは最初からキャリブレーション方式を使えば原因究明は不要）
- `is_bidirectional` フィールドはこのmakeobjバージョン(60.11)では未定義（警告になるだけで無害）
- 8方向のうちNE/SW(またはNW/SE)方向は車両が画面奥/手前を向くため細長い「マッチ棒」状に
  見えるが、これは正しい方向に正しい画像が割り当たっていれば仕様通り(本物のpak128車両も同様)

## 次回への引き継ぎ
- 新しい車両/方向系アドオンを作る際は、最初から「Z回転角ラベル付きcalib版」を作って
  実機テストする方式を採用する（机上計算は参考程度に留める）
- `render_0series.py` のDIRECTIONS命名(n/ne/e/...)は実際の世界方向と一致しないため、
  今後はZ回転角そのものをファイル名にする方が混乱がない
- 複数パーツ(球+直方体等)を組み合わせる設計は「継ぎ目」「帯のテーパー不一致」を生みやすい。
  曲面が連続する形状（流線型車両など）は最初からbmeshで1枚のロフトメッシュとして設計する方が良い
  （`create_train_shell`関数を次回のベース実装として再利用可能）
- vehicle画像はmakeobjの自動クロップでオフセットが画像ごとに変わりうる。8方向の見た目を
  そろえたい場合は4隅に不透明1pxマーカーを置いて強制的にフルキャンバス化するのが手堅い
- 0系特有の特徴（涙滴型というよりずんぐりした団子鼻、運転席窓、屋根の傾斜変化、台車・床下機器の
  作り込み）は今回未達。次フェーズで取り組む場合は参照画像を用意してシルエットを定量比較する
  （ユーザー提案の「期待vs現状の差分レポート」方式が有効）
