# tilecutterによるマルチタイル建物の自動分割

作業日: 2026-07-07

## 目標
複数タイル（2×2等）にまたがる実在建物（PLATEAU由来）を、[tilecutter](https://github.com/An-dz/tilecutter)
を使って自動的にタイル単位へ分割できるか検証する。画像生成は既存の
[plateau_building](../plateau_building/) の変換フローを流用し、ゲーム内で
1×1・2×2相当の建物としてpak化するところまで確認する。
- step1: 2×2タイル建物の自動分割・pak化・ゲーム内表示確認（タイル換算値は仮の60m/タイル）
- step2: 実在アドオン・不動産統計を根拠にタイル換算値を18m/タイルへ補正し、再検証する
- step3: 18m版がゲーム内で「タイルを大きくはみ出す」と判明 → レンダリングのズームイン
  バグ（`ortho_scale`未スケール）を発見・修正し、再生成する
- step4: step3修正後、今度は「高さ方向が見切れる」と判明 → tilecutterの各タイル抽出
  ウィンドウの構造を再調査し、Blenderの`sensor_fit`設定の誤り・タイルグリッド上での
  建物の向き/余白不足という2つの原因を切り分けて対処する
- step5: 「テクスチャ解像度が低め」という指摘を受け、ソース解像度の限界とスケール
  較正のトレードオフを整理した上で、スーパーサンプリングによる改善を試す
- step6: マルチタイル建物での四方位対応（BackImage[0]〜[3]に異なる画像）を試す
  （[plateau_building](../plateau_building/)の単体タイル四方位検証の続き）

## 結果
達成。tilecutter経由の自動分割・pak化・スケール較正(step2)・ズームインバグ修正(step3)・
接地/高さ方向のクリッピング修正(step4)のすべてを**ゲーム内表示で確認できた**
（ユーザーより「どちらも表示OK」と確認）。step5のスーパーサンプリングもゲーム内で
確認され、`53394515`（ソーステクスチャ1024×1024）は**画質十分**、`53393597`
（ソーステクスチャ128×32〜512×256）は**改善はしたが難ありな見た目**という結果に
なった。この差はスーパーサンプリングでは埋まらないソース側の解像度差に起因すると
判断でき、**「PLATEAU LOD2建物のソーステクスチャ解像度が概ね1024×1024程度あるか」を、
アドオン化に足る画質かどうかの実用的な判断基準にできそう**という結論に至った。
step6（マルチタイル建物の四方位対応）は**未解決のまま区切った**。レイアウト
入れ替え・再センタリング修正の2つの原因を特定・修正したが、ゲーム内では解消せず、
`gebaeude.cc`調査でさらに深い（タイルオフセット自体の座標変換を伴う）仕様が
関わっていることまでは分かったが、確証の持てる修正案には至らなかった
（詳細は「試したこと」参照）。単体タイルでの四方位対応・マルチタイルでの単一方向は
どちらも問題なく動作している。

## 試したこと

### tilecutterの調査
- tilecutter公式README・ソース(`main.py`/`tc.py`/`project.py`/`tcp.py`)をGitHub API経由で
  直読みして調査した
- `.tcp`プロジェクトファイルはJSON形式（`{"type":"TCP_JSON","version":"X.Y.Z","data":{...props}}`）で、
  `props`の構造（`images[方向4][季節5][フレーム][レイヤー2]`、`dims`、`files`、`dat.dat_lump`)が
  ソースから正確に判明した
- `main.py`は`-c`オプションでGUIなしのCLIモードをサポートしているが、**実際に動かすと
  `AttributeError: 'Project' object has no attribute 'pngfile'`で落ちる**ことを発見した。
  CLI引数処理コード(`main.py`)が新しい`project.Project`クラス（`pngfile_location()`等の
  メソッド名）ではなく、旧`tcproject.Project`クラス（`pngfile()`等）向けの古いAPI呼び出しの
  ままになっている、tilecutter自体の実装バグ
- そのため`main.py`のCLIグルーコードは使わず、`project.Project`と`tc.export_cutter`/
  `tc.export_writer`を直接importして呼び出す方式（[cut_with_tilecutter.py](cut_with_tilecutter.py)）に
  切り替えた。これも「tilecutterを使う」ことに変わりはなく、GUIやCLIの薄い皮を剥いで
  中身のカット処理エンジンだけを利用する形

### 環境構築
- `pip install wxPython numpy`（wxPython 4.2.5はPython 3.14向けwheelが存在し、
  headless(`wx.App(False)`)でも問題なく動作した）
- `refs/tilecutter`としてリポジトリをclone（`refs/`は既存の`.gitignore`ルールで
  対象外のため、参照ツールとして同じ場所に配置）
- `config.path_to_makeobj`は**bareコマンド名（`"makeobj"`）ではなく絶対パス
  （`C:\bin\makeobj.exe`）を渡す必要がある**と判明。`tc.py`の`export_writer`内で
  `paths.join_paths(os.getcwd(), config.path_to_makeobj)`のように**カレントディレクトリと
  結合**する実装になっており、PATH解決は行われないため

### 実データでの検証(2×2)
- 素材: [plateau_building](../plateau_building/)と同じ実データ取得フロー（`remotezip`で
  G空間情報センターのOBJ zipから直接抜き出し）で、フットプリント75.5m×95.2m・高さ30.9mの
  LOD2建物（メッシュコード`53393597`、タワー+低層棟の複合形状）を選定
- **タイル換算スケールの決定**: 単体建物(plateau_building)は「建物ごとにキャンバス一杯へ
  自動フィット」だったが、マルチタイルではタイル間の位置合わせが必要なため固定の
  実寸/タイル換算値が要る。`plateau_building`の基準建物（実寸59m）を1タイル相当とみなし、
  `TILE_SIDE_M = 60m`を採用（暫定値、pak128の実スケールとの整合は末尾の課題を参照）
- **tilecutterが前提とする座標系への合わせ込み**: tilecutterの`tile_to_screen()`(`tc.py`)は、
  入力画像の中で「タイルグリッド全体の中心が画像のどこに来るか」を厳密に前提とする。
  この対応点を、Blenderカメラの`target_z`（ワールドZ座標）から解析的に逆算する式を導出した:
  ```
  ortho_scale = TILE_SIDE_M * √2
  px_per_unit = (128 / ortho_scale) * cos(30°)
  avg_preflip = (DIMS_X + DIMS_Y + 2) * (128/8) + 128/2   # tile_to_screen の4隅平均(左上基準)
  grid_center_row = (height - avg_preflip) + 128/2         # 反転後・タイル中心へ補正
  target_z = (grid_center_row - height/2) / px_per_unit
  ```
  導出過程で2回計算ミスをして（係数`p/4`と`p/8`の取り違え、タイル中心への`+p/2`補正漏れ）
  レンダリングが空白/画角外になり、実際の出力画像を見ながら修正して最終的な式にたどり着いた
  （※上記`ortho_scale`の式は後のstep3で判明するズームインバグを内包しており、この
  2×2の時点でも実際には軽微に影響していたと考えられる。ただし2×2程度では見た目上
  「非正方形な建物によるはみ出し」と区別がつかず、この段階では問題として認識できなかった）
- 4枚に分割された画像を個別に書き出して目視確認 → 建物のタワー部分・低層棟部分が
  それぞれ別のタイルに正しく振り分けられていることを確認できた
- `dat_linter lint`・`makeobj pak128`とも問題なく通過（`dims=2,2,1`と
  `BackImage[方向][x][y][z][frame][season]=file.row.col`が自動生成される6インデックス形式。
  既存の5インデックス形式`BackImage[l][y][x][h][p]`とは異なるが、`knowledge.md`記載の
  「5形式・6形式どちらも有効」という知見の通り問題なく通った）

### 1×1相当の生成（比較用）
- 同じ`render_multitile.py`/`cut_with_tilecutter.py`をdims=1,1で実行し、
  [plateau_building](../plateau_building/)の基準建物(53394515)を単体タイルとして
  変換できることも確認した
- makeobjのデバッグ出力を比較すると、tilecutter経由の1×1建物は
  `Off X=0 Y=32 Width=128 Height=82`と、幅は128pxフル・高さも128中82pxを保持しており、
  plateau_building側で手動対応が必要だった「小さくクロップされて接地アンカーがズレる」
  問題（46×26px相当まで縮んでいた）が**tilecutter経由では最初から起きにくい**ことが
  分かった。tilecutterのマスクが常にタイルの菱形形状をベースに切り出すため、
  makeobj側の自動クロップが極端に小さくならない

### ゲーム内表示確認(step1)
- 2×2版(`TILE_SIDE_M=60m`)をゲーム内にロードし、タイル分割が正常に機能していることを
  ユーザーが確認した。元の建物（タワー+低層棟）は正方形ではないため、タイル境界から
  はみ出す部分はあるが、それはモデル形状に起因するものでタイル分割処理自体の問題ではない

### タイル換算値の補正(step2)
- `TILE_SIDE_M=60m`は根拠のない仮置きだったため、実データを基に補正した:
  - Simutrans Addon Portal（ローカルMCP経由で検索）で実際のpak128.japan建物アドオン
    「[日本風マンション詰め合わせセット](http://localhost:1080/users/54/japanese-mansion-set-shi)」
    のDatファイルを確認したところ、マンション9タイプのDimsが`1x1`〜`4x3`まで幅広く
    使われていた（`Mansion3`/`Mansion6`が`1x1`、`Mansion4`が`4x3`など）。ただし作者が
    意図した実寸メートルの記載は無し
  - 不動産統計（賃貸マンションの敷地面積は200〜500㎡が最多、目安330㎡）から、建ぺい率
    60〜80%を仮定すると建築面積は概ね230㎡前後 → 1辺**約15m四方**と逆算できた
  - 雑居ビル（狭小地が多く1棟40〜80㎡程度）を4棟束ねる場合も同程度のオーダーになり、
    両方の目安が15〜20m/タイルに収束したため、その中間値として`TILE_SIDE_M=18.0m`を採用
  - 同じ2棟を18m/タイルで再生成した結果、Dimsは
    - `53393597`（タワー+低層棟、75.5m×95.2m）: 2×2 → **5×6**
    - `53394515`（59m×46m）: 1×1 → **4×3**（奇しくも上記マンションアドオンの`Mansion4`と
      同じDims）に変化した
  - 18mへの補正後もtilecutterでの分割・makeobjビルドは問題なく成功（5×6・4×3とも
    `dat_linter lint`通過、`.pak`ビルド成功）

### カメラのズーム倍率バグの発見・修正(step3)
- 18m版をゲーム内にロードしたところ、**2棟ともタイルを大きくはみ出す**という
  報告を受けた。形状が非正方形なことによる軽微なはみ出し（step1で確認済み・許容範囲）
  とは明らかに程度が違ったため、レンダリング側のバグを疑って調査した
- 根本原因: `render_multitile.py`で`cam.data.ortho_scale`を`TILE_SIDE_M*√2`という
  **タイル数に依らない固定値**のまま、キャンバス解像度(`resolution_x/y`)だけを
  `(DIMS_X+DIMS_Y)*64`のようにタイル数に応じて拡大していた。Blenderの正投影カメラは
  `ortho_scale`が「解像度の大きい方の辺に写る実寸の world 幅」を決める仕様のため、
  解像度だけ増やして`ortho_scale`を据え置くと**視野は広がらずズームインし続ける**
  （画素密度が上がるだけ）。この結果、実際にはタイル数に関係なく常に「1タイル分の
  実寸(25.5m)」しか画角に収まらないまま、より高解像度で建物の一部を極端に拡大した
  状態になっており、タイルに割り当てられる実際の見た目サイズが常に大きくなりすぎていた
- 修正: `ortho_scale = TILE_SIDE_M*√2 * (max(width,height)/128)` として、解像度の
  拡大率に比例して`ortho_scale`（画角の実寸幅）も拡大するように変更。これにより
  「1タイル=128px=TILE_SIDE_M四方」の密度をタイル数に依らず一定に保てる
- 修正後に同じ2棟（5×6・4×3）を再レンダリングし、レンダー画像を目視で確認したところ
  建物全体が余白付きでキャンバスに収まるようになった（修正前は画面いっぱいに極端に
  拡大された建物の一部しか写っていなかった）。`dat_linter lint`・`makeobj`ビルドも
  再度成功。ゲーム内での再確認はユーザー側で実施予定

### 高さ方向のクリッピング修正(step4)
- step3修正版をゲーム内にロードしたところ、スケールは正しいが「高さ方向が見切れる」
  （`singletile_test`は左上、`multitile_test`は中央上全体的）という報告を受けた
- 検証用に[check_headroom.py](check_headroom.py)を新規作成した。tc.pyの
  `tile_to_screen()`と同じ式で各タイルの抽出矩形を再現し、画像の各ピクセル列について
  「その列をカバーするタイルのうち最も上端が高い(headroomが大きい)ものの上端」より
  上に不透明ピクセルが無いかを機械的にチェックするツール
- 原因は2つ複合していたと判明:
  1. **`ortho_scale`のsensor_fit問題**: Blenderの正投影カメラはデフォルト
     (`sensor_fit='AUTO'`)だと`ortho_scale`が「解像度の大きい方の辺」に適用される。
     建物の高さ分の余白で`height`が`width`を超えると、基準軸が幅から高さへ
     知らないうちに入れ替わり、ズーム率が意図せずズレる。`sensor_fit='HORIZONTAL'`で
     常に幅基準に固定するよう修正した
  2. **キャンバスの余白(margin)を増やしても見切れは解消しない**: `target_z`を
     `height`から連動して算出する現在の式では、キャンバスの高さを増やすと
     「タイル抽出ウィンドウの位置」と「建物の描画位置」が同じ量だけ一緒に下へ
     ずれてしまい、両者の相対的なギャップ(=見切れの有無)は変化しないことを
     数式的に確認した（実際に余白を3倍近くまで増やしても`check_headroom.py`の
     クリップ量が1pxも変わらないことで実証）
  3. 実際に効果があったのは**タイル数(Dims)を1〜2段階余分に増やす**ことと
     **建物をZ軸回転させ、背の高い部分がグリッドの奥側（ヘッドルームの大きい側）に
     来るようにする**ことの組み合わせだった。`render_multitile.py`に
     `Z_ROTATION_DEG`引数を追加し、0°/90°/180°/270°を試して
     `check_headroom.py`でクリップが解消する組み合わせを探した
  4. 最終的に採用したパラメータ: `53393597`はDims`6×6`(footprint基準の5×6から
     Y+1・X+1)・回転180°、`53394515`はDims`5×4`(footprint基準の4×3から
     X+1・Y+1)・回転270°で、どちらも`check_headroom.py`で「クリップなし」を確認
- この経験から、「タイル数(グリッドサイズ)」と「キャンバスの余白(margin)」は
  見た目には似ているが全く別のパラメータであり、後者は今回の座標系設計では
  高さ方向のクリッピングに対して無力であるという、再利用可能な知見が得られた

### テクスチャ解像度の確認とスーパーサンプリング(step5)
- ゲーム内確認後、「テクスチャの解像度が低め」という指摘を受けた。まずソースの
  PLATEAUテクスチャ自体の解像度を確認したところ、`53393597`は128×32〜512×256程度、
  `53394515`は1024×1024と、**建物によってソース解像度に大きな差がある**ことが分かった
  （低い方は実データそのものの限界で、後処理では補えない）
- 加えて、現在の設計は`TILE_SIDE_M=18m`で1タイルあたりの実寸密度(px/m)を固定して
  いるため、密度自体を上げる（=タイル数を増やす）とstep2で決めたスケール較正が崩れる
  というトレードオフがあることを説明した
- ソース解像度・スケール較正はそのままに改善できる余地として、**スーパーサンプリング**
  （2倍の解像度でレンダリングしてから高品質縮小(LANCZOS)する）を試すことにした。
  `render_multitile.py`に`SUPERSAMPLE=2`定数を追加し、レンダリング解像度を
  `width*2, height*2`にした上で、新規作成した
  [downscale_supersample.py](downscale_supersample.py)（Blender本体のPythonには
  Pillowが無いため別プロセスで実行）で最終サイズへ縮小してからtilecutterに渡す
  ようにした
- 縮小後も`check_headroom.py`で「クリップなし」を再確認し、`dat_linter lint`・
  `makeobj`ビルドも成功。ジャギー/モアレの低減が期待できるが、ソースにない
  detail が増えるわけではないため改善量は限定的
- ゲーム内で確認した結果、`53394515`（ソース1024×1024）は画質十分、`53393597`
  （ソース128×32〜512×256）はスーパーサンプリングで改善したものの見た目に難あり、
  という評価だった。ユーザーの結論: **ソーステクスチャ解像度が概ね1024×1024程度
  あるかどうかが、アドオン化に足る画質の実用的な閾値と考えてよさそう**

### マルチタイル建物の四方位対応(step6)

- [plateau_building](../plateau_building/)の単体タイル建物では四方位（BackImage[0]〜[3]に
  異なる画像）のキャリブレーションが完了していたが、マルチタイル建物では未検証だった
- tilecutterのソース(`tc.py`)を再確認したところ、`export_cutter`内に
  `if dims[3] in [1, 3]: dims = (dims[1], dims[0], dims[2])`という処理があり、
  **方向インデックス1・3（4方向のうち2つ）は内部的にX/Y次元を入れ替えて切り出す**
  仕様だと判明した（90°回転した見た目では、footprintの縦横比も画面上で入れ替わる
  ため、当然の設計）。`export_writer`側の出力ループも同様にd∈[1,3]で
  `xx=ydims, yy=xdims`と入れ替えており、`BackImage[1][x][y]`/`BackImage[3][x][y]`の
  x,yレンジがd=0,2とは入れ替わる
- 最初、非正方形の建物(`53394515`, Dims 5×4)で方向ごとに単純にDims/回転角を
  入れ替えて試したところ、方向1相当のレンダー(Dims入れ替え後の4×5)で
  300px超という大きなクリッピングが発生した。原因を確認したところ、
  「footprintに応じた自然なDims」は回転角ごとに変わる（0°/180°では4×3、
  90°/270°では3×4相当）にもかかわらず、単純に「基準方向のDimsをそのまま
  入れ替えただけ」の値を使っており、そもそも自然なDimsからズレていたため
  だった。この建物での方向ごとの正しいDims/回転の組み合わせ探索は今回は
  行わず、**正方形（`53393597`、7×7）を使うことで入れ替えの影響を実質的に
  無効化し、複雑さを避けて基本動作の検証に絞った**
- `53393597`（7×7、footprintに対して1段階余分にパディング済み）で
  0°/90°/180°/270°の4枚をレンダリングし、[check_headroom.py](check_headroom.py)で
  4枚とも「クリップなし」を確認。[cut_with_tilecutter.py](cut_with_tilecutter.py)を
  1方向専用から4方向対応（`images`を1枚または4枚受け取れるよう変更、
  `dims.directions`を実際の枚数に合わせて設定）に拡張し、
  `dims=7,7,4`・`BackImage[0]`〜`[3]`それぞれ49枚(7×7)ずつ、
  計196枚の画像を含むdat/pak生成に成功
- `dat_linter lint`・`makeobj`ビルドとも成功
- **1回目のゲーム内確認で「マップを回転すると崩れる」と判明**: 設置時（回転前）の
  向きは正常だったが、マップを90°/180°/270°回転させると表示画像と位置が
  かみ合わなくなった。単体タイルでは起きなかった問題（単体タイルは1枚絵の
  差し替えだけで済むが、マルチタイルは隣接タイル同士の内容の整合性が要るため
  露呈したと考えられる）
- 原因調査のため`refs/simutrans/src/simutrans/obj/gebaeude.cc`の
  `rotate90()`を確認したところ、`layout = (layout+3) % building_desc->get_all_layouts() & 3`
  という式（`+3 mod 4`は`-1 mod 4`と同じ）を発見。これは「マップを1回回転させると
  layout番号が1減る」という意味で、逆に読むと**layout 3 = layout 0を+90°回転した状態、
  layout 1 = layout 0を+270°回転した状態**のはずだと判明した
  （`building_desc_t::get_size()`/`get_x()`/`get_y()`の`layout & 1`判定で奇数layout
  のみX/Y入れ替えが起きる実装も確認し、tilecutterの出力仕様と一致することを確認済み）
- 検証時は素朴に`BackImage[1]`=+90°回転レンダー・`BackImage[3]`=+270°回転レンダーを
  割り当てていたが、上記の解析が正しければ**1と3が逆**だったことになる。
  `BackImage[1]`=+270°回転・`BackImage[3]`=+90°回転に入れ替えて再ビルド

### 再センタリングのバグ発見・修正(step6続き)

- layout入れ替え版を確認してもらったところ「方角によってタイル全体の中心位置から
  ずれて表示され、はみ出た部分が非表示になる」と判明。ユーザーからの「建物が単純な
  四角形でなく複雑な形状だからでは」という指摘が的確だった
- `render_multitile.py`を確認したところ、**Z回転を適用した「後」に再計算した
  バウンディングボックス中心で再センタリング**していたことが根本原因と判明。
  対称な形状なら回転前後で中心が変わらないため問題にならないが、
  `53393597`（タワー+低層棟の非対称な複合形状）のような建物では、90°回転すると
  バウンディングボックス自体の中心が実際の建物の物理的な基準点からズレるため、
  4方向それぞれが別々の基準点で再センタリングされてしまい、方向間で内容が
  食い違っていた
- 修正: `bpy.ops.object.origin_set(type='ORIGIN_GEOMETRY', center='BOUNDS')`で
  オブジェクトの原点を**回転前に一度だけ**バウンディングボックス中心へ移動させ、
  その後の回転はこの固定された原点を軸に行われるようにした。原点が既に
  バウンディングボックス中心なので、回転後は`location.x/y`を単純に0にするだけで
  常に同じ物理的基準点が世界原点に来るようになる（回転ごとに中心を再計算しない）
- 修正後、0°/90°/180°/270°すべてで[check_headroom.py](check_headroom.py)の
  クリップチェックが「OKすべて」になった（修正前は7×7で180°のみOKで、他は
  10〜30px程度のクリップが残っていた）。副次的にヘッドルーム問題も改善したのは、
  4方向の基準点が揃ったことで全体の配置がより一貫した状態になったためと考えられる
- `dat_linter lint`・`makeobj`ビルドとも成功

### 再センタリング修正後も未解決、深追いを止めた判断(step6まとめ)

- 再センタリング修正版をゲーム内で確認してもらったが、**「直ってない」**という結果
  だった。レイアウト入れ替え・再センタリング修正の2つを試しても解消しなかったことになる
- さらに`gebaeude.cc`の`rotate90()`を読み込むと、見落としていた要素があった:
  ```cpp
  new_offset = koord(building_desc->get_y(tile->get_layout()) - 1 - new_offset.y, new_offset.x);
  ```
  マップを1回転させると、**レイアウト番号だけでなく、建物内でのタイルオフセット(x,y)
  自体もこの式で変換される**。つまり「4方向をレンダリングして向きを変えるだけ」では
  不十分で、方向ごとにどのタイル位置がどのタイル位置に対応するかも、この変換式と
  正確に整合していなければならない、という当初の想定より踏み込んだ仕様だった
- tilecutter自身にはこの変換ロジックを検証する単体テストが無く（`project_test.py`は
  データ入出力のバリデーションのみで、`tile_to_screen`や回転周りの幾何計算は
  一切テストされていない）、tilecutter側の実装がこの変換と正しく整合しているかは
  ソースを読む限りでは確証が持てなかった
- ここまで3ラウンド（レイアウト入れ替え、再センタリング、根本原因の再調査）試みたが
  解消せず、**「マルチタイル建物の四方位対応」は未解決のまま今回のtry-outを終了する**
  というユーザー判断で区切った。これ以上の追求には、実際に動作している他作者の
  マルチタイル4方向建物のdatを参考にするか、`get_tile`/`rotate90`の変換式に
  正確に合わせて手動でBackImageのx,yを組み替える、といったより踏み込んだ
  リバースエンジニアリングが必要になりそうというのが現時点の見立て

## 得られた知見や失敗
- **回転を伴うレンダリングは「回転前に基準点を固定」しないと、非対称形状で
  方向ごとに中心がズレる**: `location.x -= center_x`のように毎回バウンディング
  ボックスから中心を再計算して再センタリングする実装は、対称な形状（station_test
  の立方体や単一方向のみのplateau_building）では問題にならないが、複数方向を
  レンダリングし後で1つのタイルグリッドとして結合する用途では、回転「後」に
  中心を再計算すると非対称形状では方向ごとに基準点がズレて内容が食い違う。
  `bpy.ops.object.origin_set(type='ORIGIN_GEOMETRY', center='BOUNDS')`で回転前に
  原点をバウンディングボックス中心へ固定してから回転させることで解決した。
  「レンダリング結果を目視するだけでは気づけず、複数方向を実際に組み合わせて
  初めて症状が出る」性質のバグだったため、ユーザーが実機で確認しなければ
  見つからなかった
- **マルチタイル建物の四方位対応は、非正方形だと方向ごとにDims/回転の再探索が必要**:
  tilecutterは方向1・3で内部的にX/Y次元を入れ替えて切り出す仕様（90°回転で
  footprintの縦横比が画面上で入れ替わるのに対応するため）。このため非正方形の
  建物では、方向ごとに「その回転角における自然なfootprint」が異なり、
  基準方向のDimsを単純に入れ替えるだけでは正しいパディング量にならず
  クリッピングが発生する。正方形の建物ならこの入れ替えが実質no-opになるため
  検証がシンプルになる（今回はこちらで基本動作のみ確認し、非正方形建物の
  完全な4方向対応は未着手のまま残した）
- **PLATEAU建物選定時の実用的な画質閾値: ソーステクスチャが概ね1024×1024程度あるか**:
  LOD2建物でもテクスチャ解像度は建物ごとにバラつきが大きい（今回の2棟だけでも
  128×32〜512×256 vs 1024×1024という差があった）。スーパーサンプリング等の後処理では
  この差は埋まらないため、量産・自動選定パイプラインを組む場合は、変換前にCityGML/OBJの
  テクスチャ画像サイズを機械的にチェックし、一定解像度（目安1024×1024）に満たない
  建物は除外する、というスクリーニングが有効そう
- **Blenderの正投影カメラで解像度を変えるなら`ortho_scale`も比例させる**: `ortho_scale`は
  「解像度の大きい方の辺に写る実寸幅」を決める値であり、解像度(px)だけ変えても画角
  （写る実寸範囲）は変わらない。タイル数に応じてキャンバスを大きくする設計では、
  `ortho_scale`も`解像度の拡大率`ぶん一緒にスケールさせないと、実際には常に一定の
  実寸範囲だけを高解像度で写し続ける「ズームインし続けるバグ」になる。今回は
  「タイルを大きくはみ出す」という実機フィードバックがなければ見た目の破綻を
  区別しづらい程度の誤差だったため、実際にゲームに読み込んで確認する重要性を再認識した
- **`sensor_fit`を明示しないと、高さ方向の余白追加で基準軸が入れ替わる**:
  `sensor_fit='AUTO'`のまま建物の高さ分の余白で`height`が`width`を超えると、
  `ortho_scale`の基準がいつの間にか高さ側に切り替わりズームがズレる。
  マルチタイル用途では`sensor_fit='HORIZONTAL'`を明示し、常に幅基準に固定するべき
- **「キャンバスの余白を増やす」は高さ方向のクリッピング対策にならないことがある**:
  タイル抽出ウィンドウの位置を`height`から算出する設計だと、余白を増やしても
  ウィンドウと建物の描画位置が同じ量だけ一緒にずれるだけで、両者の相対的なギャップは
  変わらない（数式的に確認済み、実際に余白を3倍にしてもクリップ量が不変なことで実証）。
  有効だったのは「タイル数(グリッドサイズ)を増やす」「建物を回転して背の高い部分を
  ヘッドルームの大きいグリッド奥側に寄せる」の2つで、どちらも
  [check_headroom.py](check_headroom.py)による機械的な検証で確認しながら試行錯誤した
- **tilecutterのCLIエントリポイントは壊れている**: `main.py -c`は`project.Project`と
  `pngfile()`等の旧API不整合でクラッシュする。README・issueにもこの点の記載は無く、
  ソースを実際に動かして初めて判明した。CLIを使いたい場合は`project.Project`+
  `tc.export_cutter`/`tc.export_writer`を直接呼ぶ方式が確実
- **`config.path_to_makeobj`はフルパス必須**: `os.path.join(cwd, path_to_makeobj)`という
  実装のため、PATHが通っていてもbareコマンド名では見つからない
- **`.tcp`のJSON構造はネストを1段でも間違えると静かに空の建物にフォールバックする**:
  validatorが型チェックに失敗すると警告ログを出すだけでデフォルト値（空画像）にフォール
  バックし、`.dat`/`.pak`自体は「エラーなく」生成されてしまう。実際に「画像が真っ白」という
  症状で気づいた。`makeobj`の「エラーが出ない＝正しいわけではない」という既知の教訓
  （`knowledge.md`）が、tilecutter側にも同様に当てはまることが分かった
- **マルチタイル用の画像スケールは「auto-fit」ではなく固定換算値が必要**: 単体建物のような
  「建物ごとにキャンバス一杯へフィット」は、タイル間の位置整合が壊れるため使えない。
  タイル同士の相対サイズを保つには共通の実寸/px換算基準が必須
- **tile_to_screenの座標系合わせは机上導出で辿り着けたが、2回のバグを経由した**:
  「レンダリング結果が空白になる」という具体的な失敗が式のバグを教えてくれた。
  過去の車両方向較正・接地アンカー修正と同様、最終的には実際の出力画像を見て
  デバッグする姿勢が有効だった
- **タイル換算値は「同一建物の実測比較」ではなく間接的な推定に留まる**: 理想は
  「同じ実在建物のPLATEAUデータと実在ベースのpak128アドオンを直接比較する」ことだったが、
  アドオン作者は著作権上の配慮からモデル元の実在建物を意図的にぼかして記載することが多く、
  直接比較できるペアは見つからなかった。代わりに「実在アドオンのDims実例」と「不動産統計
  からの逆算」という2つの独立した間接的根拠を突き合わせて`18m/タイル`という値に補正した。
  正確な較正ではなく、あくまで「違和感のない相対スケール」を狙った推定値である
- **未対応・今後の課題**:
  - ソーステクスチャ解像度が低い建物（今回で言う`53393597`程度）向けの改善策
    （超解像技術の適用、あるいは単純に「低解像度建物は変換対象から除外する」
    スクリーニングの実装）は未着手
  - 回転角・Dimsパディング量は今回2棟について試行錯誤で見つけた値であり、任意の建物に
    対して自動的に「クリップしない回転・パディング」を探索する仕組みにはなっていない
    （`check_headroom.py`はチェックのみで、自動探索ループへの組み込みは未実装）
  - `res`/`com`/`ind`（本来の市街地建物type）ではなく、station_test由来の
    `type=extension`+`waytype=track`を流用している点も[plateau_building](../plateau_building/)と同様、未解決の課題として残る
  - 非正方形の建物での四方位対応（方向ごとに異なるDims/回転の組み合わせ探索）は未着手
  - **マルチタイル建物の四方位対応（`plateau_multitile_4dir.pak`）は未解決**。
    正方形（7×7）でもゲーム内で「マップ回転時にタイル位置がかみ合わない」問題が
    解消しなかった。`gebaeude.cc`の`rotate90()`にある
    `new_offset = koord(get_y(layout)-1-new_offset.y, new_offset.x)`という
    タイルオフセット自体の座標変換にtilecutterの出力が正しく整合しているかが
    未検証で、次に着手する場合はここが最有力の調査対象
    （実在する他作者のマルチタイル4方向建物のdatと比較する、または
    この変換式に合わせて手動でBackImageのx,yを組み替える、等が候補）
